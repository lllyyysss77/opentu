# Service Worker 分阶段升级经验总结

更新日期：2026-04-23

## 现象

之前版本升级时，用户进入页面后经常会落在“新资源正在接管”的窗口里，表现为：

- 老用户刚打开页面，就开始碰新版本资源
- `Service Worker` 已经切到新版本，但新版本资源还没完全准备好
- 页面会卡在启动中间态，或者撞上旧入口引用失效 hash 的问题

这类体验最差的点在于：

- 老用户明明本地已经有一份可运行的旧版本缓存
- 却没有先稳定进入旧版本
- 而是被升级链路直接打断

## 根因

### 1. 更新版本在 install 阶段就立刻接管

之前的升级逻辑核心是：

- `install` 阶段直接 `skipWaiting()`
- `activate` 阶段直接 `clients.claim()`

这对“首次安装的新用户”是有利的，因为能更快拿到缓存能力。  
但对“已有旧版本缓存的老用户”是反效果：

- 当前会话还在跑旧版本
- 新 worker 却立刻接管页面
- 结果当前会话被硬切到尚未准备完的新版本链路

一句话：  
**首次安装和版本升级不能共用同一套 SW 接管策略。**

### 2. “已发现新版本”和“新版本已可切换”被混为一谈

之前主线程里只要看到：

- `registration.updatefound`
- 或 worker `installed`

就会倾向于提示“可以升级”。

但这只说明：

- 新 worker 已经下载好了主体脚本

不代表：

- 新版本依赖的整包静态资源已经缓存齐
- 用户点击更新后就一定能顺滑切过去

一句话：  
**worker 已安装，不等于新版本已 ready。**

## 更稳的升级模型

### 1. 新用户：最新版 + CDN 优先

新用户没有旧缓存包袱，目标是：

- 尽快进入当前最新版
- 首屏静态资源仍然走 `CDN 优先 + 源站兜底`
- SW 尽快启用缓存能力

所以首次安装仍然允许：

- `skipWaiting`
- `clients.claim`

### 2. 老用户：锁旧版本运行

老用户进入页面时，最优先目标不是升级，而是：

- 先稳定进入手头已有的旧版本
- 当前会话不要被中途切断

所以在“版本升级”场景下：

- 新 worker 安装后保持 `waiting`
- 不立刻 `skipWaiting`
- 不立刻 `clients.claim`

这样当前 tab 就能继续跑旧版本，避免进入半切换状态。

### 3. 新版本后台整包准备

等待中的新 worker 可以在后台继续做：

- 读取预缓存清单
- 拉取新版本核心静态资源
- 写入新版本对应的静态 cache

只有当这些关键资源真正准备完成后，才能认为：

- 新版本已经可以安全接管

### 4. 用户确认后再切版本

正确的交互顺序应该是：

1. 用户先用旧版本正常工作
2. 后台把新版本整包准备好
3. UI 提示“新版本已就绪”
4. 用户点“立即更新”
5. waiting worker 再 `skipWaiting`
6. 激活后 reload，切入新版本

这才是“升级提示”和“升级执行”的正确分层。

## 这次代码层面固化的规则

### 0. 升级协议必须有“待提交”和“已提交”两个阶段

只靠下面这些信号不够：

- `updatefound`
- `installed`
- `sw:newVersionReady`

因为这些信号都只能说明：

- 新 worker 出现了
- 或者新 worker 的资源大体准备好了

但还不能说明：

- 导航入口已经允许切到新版本
- 新 tab 可以安全直接吃到新版本 HTML

这次更稳的做法是引入两类版本状态：

- `committedVersion`：当前正式对页面生效的版本
- `pendingVersion`：后台已经预热完成、但尚未被用户确认切换的版本

再配一个显式状态机：

- `idle`
- `prewarming`
- `ready`
- `committing`

一句话：  
**升级不是“发现新版本就切”，而是“先后台预热，再显式提交版本”。**

### 1. install 阶段区分首次安装和版本升级

应该先判断：

- 当前注册里是否已经存在 active worker

如果没有：

- 说明是首次安装
- 可以立即接管

如果有：

- 说明是版本升级
- 不能立刻接管当前页面

### 2. activate 阶段是否 `claim` 必须可控

`activate` 不是天然就该 `clients.claim()`。

更稳的做法是：

- 首次安装：允许 claim
- 版本升级：默认不 claim
- 只有用户确认升级后，才允许新 worker 接管页面

### 3. 更新提示只由“整包 ready”触发

不要在这些时机直接弹更新提示：

- `updatefound`
- `installed`
- `activated`

更合理的是：

- 只有后台关键静态资源预缓存完成
- 再发送 `new version ready`

也就是说，提示代表的是：

- **可以切**

而不是：

- **发现了新版本**

### 4. 更新提示不能只靠广播，必须可补读

如果提示链路只依赖一次性广播，就会有两个经典丢失场景：

- 页面还没完成 duplex channel 初始化
- 页面已经打开，但当时没有订阅对应事件

结果就是：

- 后台其实已经把新版本缓存好了
- 但用户永远看不到“立即更新”

这次更稳的默认策略应该是：

- SW 把 `pendingVersion` 持久化到本地状态
- 页面启动后主动发送 `GET_VERSION_STATE`
- 即便错过广播，只要 `pendingVersion` 还在，也能重新弹出升级提示

一句话：  
**升级提示属于状态，不属于瞬时事件。**

### 5. 新 tab 在未确认升级前必须继续跑 committed 版本

这次线上暴露得最明显的问题不是“旧 tab 升级失败”，而是：

- 老 tab 还没提示升级
- 新 tab 已经直接拿到新版本 HTML
- 新 hash 资源还在慢慢拉
- 用户感知就是打开特别慢，像资源没准备好

这说明仅仅“waiting worker 不 claim”还不够，  
还必须补一条更强的规则：

- **navigation 请求只允许命中 `committedVersion`**
- **`pendingVersion` 只负责后台预热，不负责前台入口切换**

否则就会出现：

- 老 tab 继续旧版本
- 新 tab 却偷偷吃到新版本

版本体验分裂。

一句话：  
**版本切换的边界不在 worker 安装完成，而在 committedVersion 被正式提升。**

### 6. 静态缓存元数据必须带“所属版本”，不能默认写当前 SW 版本

这次实现里还有一个很容易忽略的坑：

- SW 当前脚本版本是新版本
- 但它可能正在给旧版本 `committedVersion` 回填缓存

如果此时缓存条目头部总是写：

- `x-sw-app-version = APP_VERSION`

那旧版本缓存会被误标记成“新版本资源”，后续清理和命中判断都会乱掉。

所以缓存元数据必须显式区分：

- 当前正在运行的 SW 版本
- 当前缓存条目所属的应用版本

一句话：  
**缓存条目的版本归属要跟“服务对象版本”一致，而不是跟“执行者 SW 版本”一致。**

### 7. 首屏加载进度不能把 50% 当硬上限

启动条长期卡在 `50%` 的问题，本质不是单个请求慢，而是：

- 前半段是模拟进度
- 后半段没有足够强的真实阶段信号去接管 UI

如果模拟逻辑在 `50%` 就停住，用户就会自然理解成：

- 页面卡死了

更稳的做法是：

- 模拟进度只作为早期兜底，不做过低的硬上限
- 主线程在 SW 连接、运行时恢复、React 挂载等关键阶段主动推进进度
- App ready 时再收口到 100%

另外，`sw.js` 如果同时：

- `preload`
- 又 `register('/sw.js')`

新用户首屏就很容易看到重复请求。

一句话：  
**启动页要让真实阶段接管进度，别让模拟进度长期占着主舞台。**

## 关于版本号和缓存命名的经验

### 1. SW 缓存命名必须跟版本自动变化

缓存命名不跟版本变，会带来两类问题：

- 新旧版本静态资源混用
- 老入口拿到新资源、新入口拿到旧缓存

现在项目里比较正确的做法是：

- `APP_VERSION` 来自 `version.json`
- 构建时注入到 `sw.js`
- `CACHE_NAME` / `STATIC_CACHE_NAME` 自动带版本号

一句话：  
**缓存版本隔离应该自动依赖发版号，而不是手工维护。**

### 2. 发版时要确保版本号真的 bump 了

如果版本号没变：

- `sw.js` 内部版本不会变
- cache 名也不会变
- 浏览器就可能把这次发布当成同一版本

所以不能只“发代码”，还要保证：

- 发版号同步变化

## 适合长期固化的默认策略

- 新用户：最新版、CDN-first、SW 立即启用
- 老用户：锁当前 active 版本运行
- 更新 worker：后台整包准备，不打断当前会话
- 更新提示：只在整包 ready 后显示
- 用户确认后：再 `skipWaiting + reload`
- cache 命名：始终随版本自动变化
- 导航入口：始终只认 `committedVersion`
- 升级提示：必须支持页面启动后主动补读
- 启动条：模拟兜底，真实阶段接管

## 一句话总结

`Service Worker` 升级的核心不是“尽快接管”，而是：

- 新用户尽快接入最新版
- 老用户当前会话绝不被升级过程打断

真正稳的模型不是“发现新版本就切”，而是：

**旧版本先服务，新的版本后台准备，准备好了再让用户决定何时切换。** 🔄

## 2026-04-23：启动假卡死与工具资源拆组经验

### 现象

- 启动条会长时间卡在 `68%`
- 网络面板里看到 `tool-windows-*.js` 很慢，容易误判成“首屏核心资源没下完”
- 改完 chunk 分组后，部署检查脚本和校验脚本还在找旧的 `tool-windows-*`
- 类型定义已经换了，但局部逻辑或脚本还保留旧分组名，容易报：
  `不能将类型“..."”分配给类型“IdlePrefetchGroup”`

### 根因

#### 1. `68%` 卡住，很多时候不是功能真阻塞，而是启动条写法在制造错觉

这次真实问题不是：

- React 壳子一定在等某个大包

而是：

- HTML 启动页的模拟进度硬停在 `68%`
- 后续必须等主线程更高阶段信号才能继续推进
- 偏偏同一时段 `tool-windows` 这类延后资源又开始抢网络

于是用户的体感就变成：

- 进度卡在 `68%`
- 某个大 JS 正在慢慢下
- 所以一定是这个大 JS 把首页卡住了

一句话：  
**启动页的固定百分比上限，会把“网络慢”伪装成“首屏依赖没准备好”。**

#### 2. `tool-windows` 被过度打包，导致“点一个入口，吞一整包”

之前 `tool-windows` 里混了三类东西：

- 运行时增强：任务恢复、字体预热、资产集成
- 抽屉/WinBox：项目抽屉、工具箱、知识库工具窗
- 对话框/低频 UI：媒体库、同步、备份恢复、版本提示、性能面板、命令面板、搜索

结果是：

- 首屏一旦误触发默认 idle prefetch，就会把整个大组提早拉起
- 用户只打开一个抽屉，也会顺带背上其它完全不相关的模块

一句话：  
**按“都是工具功能”打一个大组，最终一定会反噬启动和首开交互。**

#### 3. chunk 分组名不是只改一处就结束，而是“类型、构建、测试、校验”全链路资产

这次拆组后，真正需要一起改的地方不只业务代码，还包括：

- `IdlePrefetchGroup` 类型定义
- Vite `manualChunks` / `idle-prefetch-manifest` 生成逻辑
- 页面侧触发预取与监听完成状态的逻辑
- 部署包检查脚本
- 启动入口校验脚本
- CDN fallback 测试里对 chunk 文件名的示例

如果只改前 2~3 处，就很容易出现：

- 代码能跑
- 但打包检查仍报缺少 `tool-windows-*`
- 或测试仍引用旧 chunk 名

一句话：  
**chunk 分组名属于“发布契约”，不是普通内部变量。**

### 更稳的做法

#### 1. 启动进度不要设过低硬上限

更稳的规则是：

- 早期允许模拟进度兜底
- 但不要过早卡在一个很刺眼的数字
- 最终还是由真实阶段信号或 `App ready` 收口

这次调整后的思路是：

- 从 `68%` 硬停改成更平滑的高位缓进
- 让用户感知成“还在继续进入系统”，而不是“卡死”

#### 2. 默认首屏不主动预取工具重资源

### 2026-04-23：CDN 熔断不能直接等于静态资源不可用

#### 现象

- 版本升级窗口里，`tool-windows-*.js` 这类带 hash 的静态资源会被 `sw.js` 接管
- DevTools 里能看到资源直接访问 CDN 是存在的
- 但页面侧请求仍然被 Service Worker 返回 `503`
- Cache Storage 里还能看到该资源已被某个静态缓存桶记录

#### 根因

问题不在“CDN 上有没有这个文件”，而在：

- `fetchFromCDNWithFallback()` 会先看 CDN 健康状态
- 某个 CDN 连续失败后会进入 cooldown，被 `getAvailableCDNs()` 排除
- 一旦 CDN 候选为空，运行时静态资源请求就只会回源站
- 升级窗口中，旧 hash 资源在源站很可能已经 `404`
- 于是链路变成：
  `CDN 被熔断跳过 -> 源站 miss -> SW 合成 503`

一句话：

**“CDN 暂时不健康”不等于“这个版本化静态资源此刻不可取”。**

#### 修复思路

更稳的做法不是放开全局熔断，而是：

- 正常请求继续遵守 CDN cooldown
- 只有在“本地源站也失败、马上要返回 503”这一刻
- 再做一次强制 CDN recovery probe

这样能同时满足两点：

- 平时不让异常 CDN 干扰正常流量
- 升级期遇到旧 hash 资源时，不会因为短期熔断把本来可用的 CDN 资源误杀

#### 经验

- 对 hash 静态资源来说，“可恢复探测”比“严格熔断”更重要
- SW 返回 `503` 前必须输出结构化诊断日志，否则很难区分是：
  - 源站 miss
  - CDN cooldown
  - recovery probe 失败
- 这类问题本质上不是资源部署问题，而是“升级期多源回退状态机”问题

这次最终固化成：

- 默认 `idle prefetch` 分组为空
- 首屏不主动拉工具运行时
- 只有用户真的点开知识库、抽屉、媒体库、同步、备份恢复等入口时，再按需加载对应分组

一句话：  
**首屏先保证壳子可用，再按交互路径精确拉工具资源。**

#### 3. 工具资源至少拆成“运行时 / 抽屉 / 对话框 / 命令”四层

这次比较实用的拆法是：

- `tool-runtime`
  - 任务恢复、字体预热、资产集成、运行时增强
- `tool-drawers`
  - 项目抽屉、工具箱、WinBox、工具服务
- `tool-dialogs`
  - 媒体库、同步、备份恢复、版本提示、性能面板
- `tool-commanding`
  - 命令面板、画布搜索

这样做的好处是：

- 打开知识库 / 项目抽屉时，不会顺带把媒体库和性能面板一起吞掉
- 打开媒体库 / 备份恢复时，也不必背整套工具窗逻辑

#### 4. 常驻工具条只依赖“抽屉组已可用”，不要依赖整个工具系统

页面监听 SW idle prefetch 完成状态时，也要监听最小必要组：

- 不要等整个工具系统 ready
- 只等能把常驻入口显示出来的那一组

这次最终收敛到：

- 常驻工具条只把 `tool-drawers` 当优化信号
- 即便预取信号丢失，也要有超时放行

一句话：  
**能显示入口，不等于要先准备完整工具生态。**

### 这轮适合固化的检查清单

- 这个大 chunk 真是首屏必需，还是只是恰好在首屏阶段被拉起？
- 启动页卡住，是业务 ready 没到，还是模拟进度自己停住了？
- 拆分 chunk 后，下面这些有没有一起改：
  - 类型定义
  - Vite 分组映射
  - idle manifest 默认组
  - 页面侧触发逻辑
  - SW 完成状态监听
  - 部署检查脚本
  - 启动包校验脚本
  - CDN / fallback 测试示例

### 一句话总结

这轮最值钱的经验不是“把 `tool-windows` 拆小了”，而是明确了两条规则：

- **启动假卡死，很多时候是进度条写法在误导用户**
- **chunk 分组一旦成为发布契约，重命名必须全链路同步**

后面再做启动优化时，先问两件事：

- 这是不是首屏真的必须？
- 如果把它改名/拆组，相关脚本和测试是不是也一起动了？

### 2026-04-24：页面 ready 不等于离线资源已完整

#### 现象

这次线上暴露的是一个更隐蔽的问题：

- 页面首屏已经能正常打开
- 切到离线模式后，壳子也还能进
- 但点开按需加载模块时，仍然会报资源未准备好

用户体感会很割裂：

- 一眼看上去像“离线已经可用”
- 真正进入知识库、媒体库、工具窗、富文本公式等懒模块时才炸

一句话：

**首屏可用，只说明启动资源够了，不代表离线工作集已经齐了。**

#### 这次最容易误判的地方

##### 1. 只盯懒加载 JS/CSS chunk，会漏掉真正缺失的附属资源

这次排查里，大多数懒加载 JS/CSS 其实已经在 `precache-manifest` 里。

真正的缺口主要是：

- 字体（`woff` / `woff2` / `ttf`）
- 未来可能出现的图片类附属资源（`png` / `webp` / `avif` 等）

也就是说：

- 模块代码本身可能已经缓存了
- 但模块运行时依赖的字体/图片没有提前准备
- 离线首次打开相关功能时，仍然会失败

##### 2. 页面侧预取分组名和构建产物分组名一旦漂移，SW 会“看起来在预取，实际上什么也没做”

这次还踩中了一个发布契约问题：

- 页面侧请求的是一套分组名
- `vite.config.ts` 产出的 `idle-prefetch-manifest.json` 却是另一套分组名

结果是：

- `SW_PREFETCH_GROUPS` 消息发出去了
- SW 也收到了
- 但 manifest 里根本没有这些组
- 预取流程形式上成功，实际上没有新增资源入缓存

一句话：

**预取分组名不是页面内部细节，而是构建产物契约。**

##### 3. 默认 idle-prefetch 命中的是“体验组”，不等于“离线完整组”

`tool-windows` 这类分组更偏体验优化：

- 目的是减少用户第一次点开工具时的等待
- 不是为了兜底全部离线静态依赖

如果把“工具分组预取完成”误认为“离线已经完整”，后面就一定会继续踩坑。

#### 这次最终收敛出来的规则

##### 1. `precache` 负责首屏壳子，`idle-prefetch` 负责首屏之后的补齐

当前更稳的职责划分是：

- `precache-manifest`
  - 负责启动必需资源
  - 目标是保证页面能稳定打开
- `idle-prefetch-manifest`
  - 负责页面 ready 后的空闲补齐
  - 目标是把首屏之外、但离线仍可能首次命中的静态资源补进缓存

不要试图只靠 `precache` 把所有离线资源都打满，也不要指望默认工具分组自动覆盖全部资源。

##### 2. 页面 ready 后要有一个明确的“补剩余静态资源”动作

这次最终采用的方案是新增一个专用分组：

- `offline-static-assets`

它的职责不是表达业务能力，而是表达缓存语义：

- 收集所有**未进入 `precache`** 的剩余静态资源
- 在页面真正 ready 后，由主线程主动请求 SW 空闲预取

这样做有两个好处：

- 不会额外放大首屏安装期的缓存成本
- 又能在用户真正切离线前，把非首屏但高概率命中的静态依赖补齐

##### 3. 构建侧必须以 `precache` 为基线做增量，不要重复扫同一批资源

这次补齐逻辑里，构建阶段先读取 `precache-manifest.json`，再生成：

- `offline-static-assets = 全量静态资源 - precache 已包含资源`

这样能避免两个问题：

- 重复预取已经在 install 阶段缓存过的资源
- 让“离线补齐”从语义上始终保持增量

##### 4. 对外暴露的分组名要尽量少，且要和构建口径共用同一套类型

这次最终没有继续扩散页面语义分组，而是收敛成：

- `tool-windows`
- `offline-static-assets`

经验是：

- 页面侧想表达“运行时 / 抽屉 / 对话框”的语义没问题
- 但一旦这些名字要穿过 SW 消息边界，就必须和构建产物完全一致
- 最好由共享类型统一导出，避免页面和构建各写一套字符串

#### 这轮适合固化的检查清单

- 页面 ready 时，离线补齐流程有没有真正发起？
- `idle-prefetch-manifest` 里新增分组，对 `precache` 的增量到底是几项？
- 离线失败时，缺的是：
  - JS chunk
  - CSS
  - 字体
  - 图片
  - 还是 SW 分组命名本身不匹配？
- 页面侧使用的预取分组名，是否和构建产物 manifest 完全一致？
- 新增静态资源类型时，是否同步评估：
  - 要不要进 `precache`
  - 还是应该进入 post-boot 补齐组

#### 一句话总结

这次最值钱的经验不是“又多预取了一组资源”，而是明确了：

- **页面 ready 只代表壳子 ready，不代表离线工作集 ready**
- **离线完整性要盯整个静态依赖面，而不只盯懒加载 JS**
- **SW 预取分组名一旦进入消息协议，就必须当作发布契约来维护**
