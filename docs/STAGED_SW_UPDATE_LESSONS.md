# Service Worker 分阶段升级经验总结

更新日期：2026-04-23

## 现象

之前版本升级时，用户进入页面后经常会落在“新资源正在接管”的窗口里，表现为：

- 老用户刚打开页面，就开始碰新版本资源
- `Service Worker` 已经切到新版本，但新版本资源还没完全准备好
- 页面会卡在启动中间态，或者撞上旧入口引用失效 hash 的问题

这类体验最差的点在于：

- 老用户明明本地已经有一份可运行的旧版本缓存
- 却没有先稳定进入旧版本
- 而是被升级链路直接打断

## 根因

### 1. 更新版本在 install 阶段就立刻接管

之前的升级逻辑核心是：

- `install` 阶段直接 `skipWaiting()`
- `activate` 阶段直接 `clients.claim()`

这对“首次安装的新用户”是有利的，因为能更快拿到缓存能力。  
但对“已有旧版本缓存的老用户”是反效果：

- 当前会话还在跑旧版本
- 新 worker 却立刻接管页面
- 结果当前会话被硬切到尚未准备完的新版本链路

一句话：  
**首次安装和版本升级不能共用同一套 SW 接管策略。**

### 2. “已发现新版本”和“新版本已可切换”被混为一谈

之前主线程里只要看到：

- `registration.updatefound`
- 或 worker `installed`

就会倾向于提示“可以升级”。

但这只说明：

- 新 worker 已经下载好了主体脚本

不代表：

- 新版本依赖的整包静态资源已经缓存齐
- 用户点击更新后就一定能顺滑切过去

一句话：  
**worker 已安装，不等于新版本已 ready。**

## 更稳的升级模型

### 1. 新用户：最新版 + CDN 优先

新用户没有旧缓存包袱，目标是：

- 尽快进入当前最新版
- 首屏静态资源仍然走 `CDN 优先 + 源站兜底`
- SW 尽快启用缓存能力

所以首次安装仍然允许：

- `skipWaiting`
- `clients.claim`

### 2. 老用户：锁旧版本运行

老用户进入页面时，最优先目标不是升级，而是：

- 先稳定进入手头已有的旧版本
- 当前会话不要被中途切断

所以在“版本升级”场景下：

- 新 worker 安装后保持 `waiting`
- 不立刻 `skipWaiting`
- 不立刻 `clients.claim`

这样当前 tab 就能继续跑旧版本，避免进入半切换状态。

### 3. 新版本后台整包准备

等待中的新 worker 可以在后台继续做：

- 读取预缓存清单
- 拉取新版本核心静态资源
- 写入新版本对应的静态 cache

只有当这些关键资源真正准备完成后，才能认为：

- 新版本已经可以安全接管

### 4. 用户确认后再切版本

正确的交互顺序应该是：

1. 用户先用旧版本正常工作
2. 后台把新版本整包准备好
3. UI 提示“新版本已就绪”
4. 用户点“立即更新”
5. waiting worker 再 `skipWaiting`
6. 激活后 reload，切入新版本

这才是“升级提示”和“升级执行”的正确分层。

## 这次代码层面固化的规则

### 1. install 阶段区分首次安装和版本升级

应该先判断：

- 当前注册里是否已经存在 active worker

如果没有：

- 说明是首次安装
- 可以立即接管

如果有：

- 说明是版本升级
- 不能立刻接管当前页面

### 2. activate 阶段是否 `claim` 必须可控

`activate` 不是天然就该 `clients.claim()`。

更稳的做法是：

- 首次安装：允许 claim
- 版本升级：默认不 claim
- 只有用户确认升级后，才允许新 worker 接管页面

### 3. 更新提示只由“整包 ready”触发

不要在这些时机直接弹更新提示：

- `updatefound`
- `installed`
- `activated`

更合理的是：

- 只有后台关键静态资源预缓存完成
- 再发送 `new version ready`

也就是说，提示代表的是：

- **可以切**

而不是：

- **发现了新版本**

## 关于版本号和缓存命名的经验

### 1. SW 缓存命名必须跟版本自动变化

缓存命名不跟版本变，会带来两类问题：

- 新旧版本静态资源混用
- 老入口拿到新资源、新入口拿到旧缓存

现在项目里比较正确的做法是：

- `APP_VERSION` 来自 `version.json`
- 构建时注入到 `sw.js`
- `CACHE_NAME` / `STATIC_CACHE_NAME` 自动带版本号

一句话：  
**缓存版本隔离应该自动依赖发版号，而不是手工维护。**

### 2. 发版时要确保版本号真的 bump 了

如果版本号没变：

- `sw.js` 内部版本不会变
- cache 名也不会变
- 浏览器就可能把这次发布当成同一版本

所以不能只“发代码”，还要保证：

- 发版号同步变化

## 适合长期固化的默认策略

- 新用户：最新版、CDN-first、SW 立即启用
- 老用户：锁当前 active 版本运行
- 更新 worker：后台整包准备，不打断当前会话
- 更新提示：只在整包 ready 后显示
- 用户确认后：再 `skipWaiting + reload`
- cache 命名：始终随版本自动变化

## 一句话总结

`Service Worker` 升级的核心不是“尽快接管”，而是：

- 新用户尽快接入最新版
- 老用户当前会话绝不被升级过程打断

真正稳的模型不是“发现新版本就切”，而是：

**旧版本先服务，新的版本后台准备，准备好了再让用户决定何时切换。** 🔄
